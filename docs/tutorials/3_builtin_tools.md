# 内置工具（Built-in Tools）完全指南

作为 coding agent，核心能力不仅体现在对代码逻辑的理解与生成上，更体现在与外部环境的高效交互之中。本教程将系统性地介绍当前默认 agent worker（`src/kimix/agent_worker.json`）所拥有的全部**内置工具（Built-in Tools）**，并深入讲解如何在提示词（Prompt）中精准地引导 agent 调用这些工具，以完成从文件操作、代码执行、信息检索到复杂任务编排的各类工作。

> **注意**：`agent_worker.json` 通过 `extend: default` 继承基础配置，但由于显式重写了 `tools` 字段，最终可用的工具集**仅包含** `agent_worker.json` 中列出的 14 个工具。以下列表即为实际可用的完整配置。

## 什么是内置工具

内置工具是 agent 与操作系统、文件系统、网络以及外部进程进行交互的**标准化接口**。每一次工具调用都是一次结构化的 Function Call，agent 根据当前任务的上下文自主决策是否需要调用工具、调用哪个工具以及传入哪些参数。

理解并善用这些工具，意味着你可以让 agent 做到：

- **不只"说"代码，而是直接"写"代码到文件中**（`WriteFile`、`EditFile`）
- **不只"推测"问题，而是直接"读"文件、"搜"代码来验证**（`ReadFile`、`Grep`、`Glob`）
- **不只"给出命令"，而是直接"运行"并观察输出**（`Run`）
- **处理耗时任务时进行异步管理**（`TaskOutput`、`Input`）
- **将外部资料纳入工作流**（`FetchURL`、`Search`）

## 工具全景概览

当前内置工具可分为以下六大类：

| 分类 | 包含工具 | 典型用途 |
| :--- | :--- | :--- |
| **文件与 I/O** | `WriteFile`, `ReadFile`, `EditFile`, `Glob`, `Grep` | 创建、读取、修改、搜索文件与目录 |
| **代码执行** | `Run`, `Python` | 执行可执行文件、bash 命令或 Python 代码 |
| **进程管理** | `TaskOutput`, `Input` | 与后台进程交互、获取输出、发送输入 |
| **搜索与信息** | `FetchURL`, `Search` | 获取网页内容、搜索本地 skill |
| **状态与标志** | `SetTodoList`, `StepMemory` | 追踪多步骤任务进度、持久化工作步骤（可跨 compact 恢复） |
| **子代理** | `Agent` | 委派子任务给独立 agent |

## 核心工具详解

### 一、文件与 I/O 操作

这是 coding agent 最高频使用的工具集合，直接决定了 agent 能否安全、精准地修改项目代码。

#### `WriteFile` —— 写入文件
- **功能**：将内容写入指定文件，支持 `overwrite`（覆盖，默认）和 `append`（追加）两种模式。
- **要点**：当内容超过 100 行时，规范建议拆分为多次调用，第一次用 `overwrite`，后续用 `append`。
- **示例场景**：生成全新的源码文件、配置文件或 README。

#### `ReadFile` —— 读取文件
- **功能**：按行读取文本文件，支持指定起始行（`line_offset`）和读取行数（`n_lines`），也支持从末尾倒数读取（负数 `line_offset`）。
- **要点**：单行长度过长时会自动截断；对于大文件，应分块读取以避免上下文爆炸。
- **示例场景**：查看现有代码结构、读取日志文件、核对配置文件内容。

#### `EditFile` —— 精准替换
- **功能**：在文本文件中进行字符串级别的批量替换，支持单行或多行内容替换，也支持替换所有匹配项（`replace_all`）。
- **要点**：这是执行**最小 diff 修改**的首选方式。相比直接覆盖整个文件，它能最大程度保留原有格式、注释和空白行，减少不必要的变更。
- **示例场景**：修改变量名、插入几行新代码、更新配置参数。

#### `Glob` —— 文件模式匹配
- **功能**：基于通配符（如 `*`, `?`）查找文件，可指定是否包含目录。
- **要点**：避免使用前缀 `**` 或在过大目录中执行，以防性能问题。
- **示例场景**：快速列出所有 `.py` 文件、查找测试文件目录结构。

#### `Grep` —— 内容级搜索
- **功能**：基于正则表达式在文件内容中搜索，功能相当于 `ripgrep`。
- **高级选项**：支持大小写不敏感（`-i`）、多行模式（`multiline`）、显示上下文行（`-B`/`-A`/`-C`）、文件类型过滤（`type`/`glob`）等。
- **示例场景**：查找某个函数在哪些文件中被调用、定位特定错误日志模式、搜索 TODO 注释。

### 二、代码执行与进程管理

#### `Run` —— 执行程序或命令
- **功能**：运行指定的可执行文件，或执行内置映射的常用 bash 命令。可传入参数（`args`）、指定工作目录（`cwd`）、捕获输出到文件（`output_path`），支持超时控制（默认 10 秒，范围 3–180 秒）。
- **内置命令映射**：`Run` 内部维护了 100+ 常用命令的映射表（如 `cat`, `ls`, `grep`, `find`, `sed`, `curl`, `wget`, `diff`, `tar`, `ps`, `top`, `df`, `du` 等），可直接通过 `Run` 调用这些命令，无需指定完整路径。
- **后台运行**：当命令执行超过 `timeout` 时，`Run` 会将其转为后台任务，并返回 `task_id`，后续可通过 `TaskOutput` 获取输出、通过 `Input` 发送输入。
- **与 Shell 的区别**：`Run` 不是通过 shell 解释器执行命令，而是直接运行可执行文件或调用内部映射的命令实现，行为更可控、更安全。
- **示例场景**：编译项目（`make`, `cmake`）、运行单元测试、调用命令行工具（`git`, `npm` 等）、执行 Python 脚本。

#### `Python` —— 执行 Python 代码
- **功能**：在独立的子进程中执行 Python 代码，直接接收代码字符串。与 `Run` 不同，`Python` 无需通过 `python -c` 间接调用。
- **参数**：
  - `code`（必需）：要执行的 Python 代码字符串
  - `output_path`（可选）：输出文件路径，若指定则将执行输出写入该文件
  - `timeout`（可选）：超时秒数，默认 10 秒，范围 3–60 秒
- **超时处理**：与 `Run` 一致，超时后自动转为后台任务，返回 `task_id` 供 `TaskOutput` / `Input` 使用
- **并发控制**：内置信号量限制，最多同时运行 8 个 Python 进程
- **长代码处理**：超过 30000 字符的代码会自动写入临时 `.py` 文件执行，避免命令行长度限制
- **示例场景**：快速验证代码逻辑、执行脚本片段、数据处理

#### `TaskOutput` —— 获取后台任务输出
- **功能**：获取指定后台任务的累计输出，支持阻塞等待（`block`）或定时轮询（`wait_time`），支持强制终止（`kill`）和导出输出（`output_path`）。
- **示例场景**：监控一个长时间运行的编译进程、查看后台测试的输出结果。

#### `Input` —— 向进程发送输入
- **功能**：向正在运行的后台进程的标准输入（stdin）发送文本。用于与交互式程序对话。
- **示例场景**：向交互式安装脚本发送确认回答、向 REPL 会话发送命令。

**典型工作流**：
1. 用 `Run` 启动一个可能耗时的命令（如 `python setup.py`）。
2. 若命令超时，`Run` 会返回 `task_id`，提示使用 `TaskOutput` 或 `Input`。
3. 用 `TaskOutput` 定期观察输出。
4. 如需交互，用 `Input` 向 `task_id` 发送指令。

### 三、信息检索与搜索

#### `FetchURL` —— 获取网页内容
- **功能**：使用无头浏览器获取网页内容并返回 Markdown 格式。适用于抓取文档、API 说明、技术博客等。
- **示例场景**：读取某个包的官方文档页面、获取 GitHub issue 的详细内容。

#### `Search` —— Skill 搜索
- **功能**：基于语义搜索在本地 skill 目录中检索与关键词相关的 skill。支持指定 `dest_path` 限制搜索目录。
- **示例场景**：查找是否有现成的 skill 可以处理当前任务、了解项目中可用的 agent 扩展能力。

---

### 四、状态与标志管理

#### `SetTodoList`
- **功能**：追踪复杂/多步骤任务的进度。支持 `pending`（待办）、`in_progress`（进行中）、`done`（完成）三种状态。
- **要点**：仅在复杂任务中使用，且每次更新必须传入**完整的任务列表**。
- **示例场景**：重构一个大型模块时，将任务拆分为"调研依赖 → 修改接口 → 更新测试 → 验证编译"并持续更新进度。

#### `StepMemory` —— 持久化工作步骤
- **功能**：将已完成的关键步骤持久化到文件（`.kimix_cache/steps/{session_id}.json`），即使上下文被 compact 后也能恢复完整历史。
- **两种操作**：
  - `action="save"`：记录一个步骤。必须提供 `step`（步骤描述），可选 `result`（结果摘要）、`files`（涉及文件列表）、`brief`（简短标题用于 compact 后快速索引）
  - `action="load"`：加载历史。支持按 `step` 文本过滤或按 `files` 搜索 ToolCallReason 记录
- **自动压缩**：超过 200 条记录时，将最旧的一半压缩为摘要（保留 `seq`、`time`、`brief`，步骤和结果标记为 `[compacted]`）
- **示例场景**：在 context compaction 后恢复之前的任务进度、跨会话追踪多阶段任务的完成情况

### 五、子代理与代码质量

#### `Agent`
- **功能**：生成一个独立的子代理来处理特定子任务。
- **示例场景**：将代码审查、文档翻译、独立模块开发等任务委派给子代理，以并行方式提升效率。

## 如何通过提示词引导工具使用

agent 是否会调用工具、调用得是否准确，很大程度上取决于提示词的表述方式。以下是几种行之有效的引导策略。

### 1. 明确指令式（Direct Instruction）

直接在提示词中指定需要调用的工具名称和目的。

> **示例**：
> "请先用 `Glob` 查找 `src/` 目录下所有的 `.cpp` 文件，然后用 `ReadFile` 逐一遍历这些文件，查找其中是否包含 `deprecated` 标记。最后将结果整理到一个新的 `report.md` 文件中，使用 `WriteFile` 写入。"

**效果**：agent 会按照你指定的工具链一步一步执行，路径清晰，可控性强。

### 2. 目标驱动式（Goal-Oriented）

不指定具体工具，而是描述期望达成的目标，让 agent 自主决策。

> **示例**：
> "我想知道这个项目的入口函数在哪里，以及它依赖了哪些第三方库。请自行搜索项目代码并给出结论。"

**效果**：agent 会自主组合 `Glob` + `Grep` + `ReadFile` 等工具进行调研。这种方式更适合探索性任务。

### 3. 约束条件式（Constrained Execution）

在提示词中加入显式约束，限制 agent 的行为边界。

> **示例**：
> "请修改 `config.py` 中的 `MAX_RETRIES` 值为 `5`。要求：
> 1. 必须使用 `EditFile` 进行最小化修改，不要覆盖整个文件；
> 2. 修改前先用 `ReadFile` 查看当前内容，确认行号；
> 3. 修改后再次用 `ReadFile` 验证结果。"

**效果**：约束条件能有效避免 agent 因"图省事"而采取不够优雅的操作（如直接覆盖文件）。

### 4. 分步工作流式（Step-by-Step Workflow）

对于复杂任务，将步骤拆解并在每一步提示中嵌入工具调用要求。

> **示例**：
> "我们需要为项目添加一个新的 CLI 命令。请按以下步骤执行：
> 1. **调研**：用 `Glob` 和 `ReadFile` 查看现有 CLI 命令的实现方式；
> 2. **实现**：编写新命令的实现代码，使用 `WriteFile` 或 `EditFile`；
> 3. **验证**：运行 `Run` 执行相关测试，确认功能正常；
> 4. **收尾**：用 `SetTodoList` 更新任务进度为全部完成。"

### 5. 元提示词式（Meta-Prompting）

如果你正在设计一个系统提示词（System Prompt），可以在其中固化 agent 使用工具的通用准则。

> **示例**（可作为 system prompt 的一部分）：
> "你是一个精通工具使用的 coding agent。在执行任务时，请遵循以下原则：
> - **先观察，后动手**：修改任何文件前，优先使用 `ReadFile` 或 `Grep` 确认上下文；
> - **最小化变更**：编辑文件时优先使用 `EditFile`，避免不必要的全文件覆盖；
> - **耗时任务异步化**：对于可能超过 10 秒的命令，使用 `Run` 并后续通过 `TaskOutput` 监控；
> - **善用子代理**：当子任务可以独立执行且不影响主流程时，使用 `Agent` 并行处理。"

## 最佳实践与典型场景

### 场景 A：安全地重构一个函数

```text
用户：请将项目中的所有 `print_error` 调用替换为 `log_error`。

Agent 应执行：
1. Grep 搜索 "print_error"，确定所有出现位置；
2. ReadFile 查看关键文件，确认是否有同名局部变量或字符串误匹配；
3. 对每个确认的文件使用 EditFile 进行替换；
4. 替换后 Grep 再次验证是否还有残留；
5. Run 运行项目测试，确认没有破坏功能。
```

### 场景 B：处理一个交互式命令

```text
用户：请帮我运行这个交互式脚本 `setup.py` 并完成安装向导。

Agent 应执行：
1. Run 启动 `python setup.py`（若超时则转为后台任务，获取 task_id）；
2. TaskOutput 观察脚本输出的提示问题；
3. Input 向进程发送对应的回答；
4. 循环步骤 2-3 直到安装完成；
5. TaskOutput 最终确认成功信息。
```

### 场景 C：引入一个需要多文件修改的新功能

```text
用户：请为这个项目添加对 JWT 认证的支持。

Agent 应执行：
1. Glob + ReadFile 调研现有认证中间件结构；
2. 撰写实现方案；
3. 按步骤修改多个文件（EditFile / WriteFile）；
4. SetTodoList 跟踪各子任务状态；
5. Run 运行测试验证。
```

### 场景 D：将外部文档纳入分析

```text
用户：请阅读附件中的设计文档，并检查代码是否实现了文档中的所有要求。

Agent 应执行：
1. 若为网页链接，使用 FetchURL 获取内容；
2. 分析内容，提取需求清单；
3. Grep + ReadFile 检查代码实现；
4. 对比并输出差异报告。
```

## Plan Mode（计划模式）

Plan Mode（`/plan`）是一种**先规划、后执行**的两阶段工作流，将"方案设计"与"代码实现"解耦：

1. **生成计划** —— 由专门的规划代理（加载 `agent_planner.json`）分析需求，并将完整的执行方案写入 Markdown 文件。
2. **执行计划** —— 用户审阅计划后，再由常规的 Worker 代理负责具体实现。

#### 使用方式

```
/plan              # 默认输出到 plan.md
/plan:roadmap.md   # 指定自定义文件路径
```

执行命令后，输入你的需求描述，以 `/end` 结束输入，或以 `/cancel` 取消。

#### 工作流程

1. **输入需求** —— 描述你要构建或重构的内容。
2. **生成计划** —— 规划代理使用 Note 工具将计划写入文件，并自动打开文件供你审阅。
3. **确认执行** —— 输入 `y` 将计划交给 Worker 代理执行，或输入 `n` 保留计划以便手动执行。

#### 适用场景

- **复杂功能开发** —— 涉及多文件重构、架构调整或新模块开发时，预先设计能显著降低返工风险。
- **先审后做** —— 在执行任何代码修改前审阅计划，便于调整范围或拆分任务。
- **职责分离** —— 规划代理专注架构设计，Worker 代理专注代码实现，提升整体质量。

## 总结

内置工具是 coding agent 从"语言模型"进化为"可执行代理"的桥梁。掌握这些工具的特性与适用场景，并在提示词中清晰、合理地引导其使用，是发挥 agent 最大效能的关键。

**核心原则回顾**：

1. **观察先于行动**：`ReadFile` / `Grep` / `Glob` 永远是修改前的第一步。
2. **最小化改动**：`EditFile` 是编辑文件的首选，`WriteFile` 仅用于创建新文件或明确需要整体重写的情况。
3. **耗时任务异步化**：善用 `Run` 的超时自动转后台机制，结合 `TaskOutput` / `Input` 管理交互与长时任务。
4. **外部信息无缝整合**：通过 `FetchURL` 将外部信息纳入分析。
5. **先规划后构建**：对于复杂任务，使用 `/plan` 将设计与实现分离。

通过刻意地在提示词中植入这些原则，你将能够让 agent 更加可靠、高效、可控地完成各类软件开发任务。
