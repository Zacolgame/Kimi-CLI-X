# 内置工具（Built-in Tools）完全指南

作为 coding agent，核心能力不仅体现在对代码逻辑的理解与生成上，更体现在与外部环境的高效交互之中。本教程将系统性地介绍当前 agent 所拥有的全部**内置工具（Built-in Tools）**，并深入讲解如何在提示词（Prompt）中精准地引导 agent 调用这些工具，以完成从文件操作、代码执行、信息检索到复杂任务编排的各类工作。

## 什么是内置工具

内置工具是 agent 与操作系统、文件系统、网络以及外部进程进行交互的**标准化接口**。每一次工具调用都是一次结构化的 Function Call，agent 根据当前任务的上下文自主决策是否需要调用工具、调用哪个工具以及传入哪些参数。

理解并善用这些工具，意味着你可以让 agent 做到：

- **不只“说”代码，而是直接“写”代码到文件中**（`WriteFile`、`StrReplaceFile`）
- **不只“推测”问题，而是直接“读”文件、“搜”代码来验证**（`ReadFile`、`Grep`、`Glob`）
- **不只“给出命令”，而是直接“运行”并观察输出**（`Run`、`Python`）
- **处理耗时任务时进行异步管理**（`TaskOutput` `Input`）
- **将 PDF、Word 等非结构化文档纳入工作流**（`Pdf2md`、`Docx2md`）
- **在复杂任务前进入计划模式，降低出错概率**（`EnterPlanMode`、`ExitPlanMode`）

## 工具全景概览

当前内置工具可分为以下八大类：

| 分类 | 包含工具 | 典型用途 |
| :--- | :--- | :--- |
| **文件与 I/O** | `WriteFile`, `ReadFile`, `StrReplaceFile`, `Mkdir`, `Rm` | 创建、读取、修改、删除文件与目录 |
| **搜索与定位** | `Glob`, `Grep` | 按模式查找文件、在文件内容中搜索 |
| **代码执行** | `Python`, `Run` | 执行 Python 脚本或任意可执行程序 |
| **进程管理** | `Input`, `TaskOutput` | 与后台进程交互、管理异步任务 |
| **状态与标志** | `SetTodoList` | 设置状态标志、追踪多步骤任务进度 |
| **文档转换** | `FetchURL`, `Pdf2md`, `Docx2md` | 获取网页内容、转换 PDF/Word 为 Markdown |
| **计划与编排** | `EnterPlanMode`, `ExitPlanMode` | 在重大修改前制定计划并让用户确认 |
| **子代理与质量** | `Agent`, `Cpplint` | 委派子任务、检查 C++ 代码语法 |

## 核心工具详解

### 一、文件与 I/O 操作

这是 coding agent 最高频使用的工具集合，直接决定了 agent 能否安全、精准地修改项目代码。

#### `WriteFile` —— 写入文件
- **功能**：将内容写入指定文件，支持 `overwrite`（覆盖，默认）和 `append`（追加）两种模式。
- **要点**：当内容超过 100 行时，规范建议拆分为多次调用，第一次用 `overwrite`，后续用 `append`。
- **示例场景**：生成全新的源码文件、配置文件或 README。

#### `ReadFile` —— 读取文件
- **功能**：按行读取文本文件，支持指定起始行（`line_offset`）和读取行数（`n_lines`），也支持从末尾倒数读取（负数 `line_offset`）。
- **要点**：单行长度过长时会自动截断；对于大文件，应分块读取以避免上下文爆炸。
- **示例场景**：查看现有代码结构、读取日志文件、核对配置文件内容。

#### `StrReplaceFile` —— 精准替换
- **功能**：在文本文件中进行字符串级别的批量替换，支持单行或多行内容替换，也支持替换所有匹配项（`replace_all`）。
- **要点**：这是执行**最小 diff 修改**的首选方式。相比直接覆盖整个文件，它能最大程度保留原有格式、注释和空白行，减少不必要的变更。
- **示例场景**：修改变量名、插入几行新代码、更新配置参数。

#### `Mkdir` / `Rm` —— 目录与文件管理
- **`Mkdir`**：递归创建目录（包括父目录）。
- **`Rm`**：删除文件或目录。
- **要点**：这两个工具权力较大，提示词中应明确是否需要创建或清理目录，避免误删关键文件。

### 二、信息检索与搜索

在动手修改代码之前，agent 必须先“看清”项目全貌。搜索类工具就是它的眼睛。

#### `Glob` —— 文件模式匹配
- **功能**：基于通配符（如 `*`, `?`）查找文件，可指定是否包含目录。
- **要点**：避免使用前缀 `**` 或在过大目录中执行，以防性能问题。
- **示例场景**：快速列出所有 `.py` 文件、查找测试文件目录结构。

#### `Grep` —— 内容级搜索
- **功能**：基于正则表达式在文件内容中搜索，功能相当于 `ripgrep`。
- **高级选项**：支持大小写不敏感（`-i`）、多行模式（`multiline`）、显示上下文行（`-B`/`-A`/`-C`）、文件类型过滤（`type`/`glob`）等。
- **示例场景**：查找某个函数在哪些文件中被调用、定位特定错误日志模式、搜索 TODO 注释。

### 三、代码执行与计算

#### `Python` —— 执行 Python 代码
- **功能**：在子进程中运行 Python 代码，支持后台运行（`run_in_background`）、超时控制（`timeout`）以及将结果输出到文件（`dest`）。
- **示例场景**：运行数据分析脚本、执行自动化测试、计算复杂数学表达式、快速验证一段代码的行为。

#### `Run` —— 执行任意程序
- **功能**：运行指定的可执行文件，可传入参数（`args`）、指定工作目录（`cwd`）、捕获输出（`output_path`），同样支持后台运行和超时控制。
- **要点**：当需要与用户输入交互时，必须设置 `run_in_background=true`，随后配合 `Input` 工具使用。
- **示例场景**：编译项目（`make`, `cmake`）、运行单元测试、启动开发服务器、调用命令行工具（`git`, `npm` 等）。

### 四、进程与任务管理

对于耗时较长的任务（如模型训练、大规模编译、交互式程序），agent 需要异步管理能力。

- **`Input`**：向正在运行的后台进程的标准输入（stdin）发送文本。用于与交互式程序对话。
- **`TaskOutput`**：列出所有后台任务及其当前状态（运行中、已完成等）。或获取指定后台任务的累计输出，可设置阻塞等待（`block`）或定时轮询（`wait_time`）。

**典型工作流**：
1. 用 `Run` 启动一个长时间运行的命令，并设置 `run_in_background=true`，获取 `task_id`。
2. 用 `TaskOutput` 定期观察输出。
3. 如需交互，用 `Input` 向 `task_id` 发送指令。

### 五、状态与标志管理

#### `SetTodoList`
- **功能**：追踪复杂/多步骤任务的进度。支持 `pending`（待办）、`in_progress`（进行中）、`done`（完成）三种状态。
- **要点**：仅在复杂任务中使用，且每次更新必须传入**完整的任务列表**。
- **示例场景**：重构一个大型模块时，将任务拆分为“调研依赖 → 修改接口 → 更新测试 → 验证编译”并持续更新进度。

### 六、文档解析与转换

在处理外部资料时，这些工具能将非结构化数据转化为 agent 可处理的 Markdown 文本。

- **`FetchURL`**：使用无头浏览器获取网页内容并返回 Markdown 格式。适用于抓取文档、API 说明、技术博客等。
- **`Pdf2md`**：将 PDF 转换为 Markdown，支持可选的 OCR（`ocr`）、图片提取（`extract_images`）、表格提取（`extract_tables`）以及指定页码（`page_range`）。
- **`Docx2md`**：将 Word 文档转换为 Markdown，保留段落、表格和基本格式。

### 七、计划模式与任务编排

#### `EnterPlanMode` / `ExitPlanMode`
- **功能**：在面对**新功能开发、架构调整、多文件改动或需求不明确**的任务时，agent 可以先进入计划模式。
- **工作流**：
  1. 调用 `EnterPlanMode`。
  2. Agent 会先进行调研，然后将计划写入文件。
  3. 调用 `ExitPlanMode` 向用户展示计划选项（通常会提供 2-3 个备选方案并标注推荐项）。
  4. 用户确认或修改后，再继续执行。
- **要点**：这是防止“盲目动手导致大范围返工”的关键机制。提示词中应明确要求 agent 在复杂场景下使用计划模式。

### 八、子代理与代码质量

#### `Agent`
- **功能**：生成一个独立的子代理来处理特定子任务。支持后台运行（`run_in_background`）和深度思考模式（`thinking`）。
- **示例场景**：将代码审查、文档翻译、独立模块开发等任务委派给子代理，以并行方式提升效率。

#### `Cpplint`
- **功能**：使用 `clangd` 验证 C++ 文件的语法和基本代码规范。
- **示例场景**：在提交 C++ 代码修改前，自动运行语法检查以确保没有引入编译错误。

## 如何通过提示词引导工具使用

agent 是否会调用工具、调用得是否准确，很大程度上取决于提示词的表述方式。以下是几种行之有效的引导策略。

### 1. 明确指令式（Direct Instruction）

直接在提示词中指定需要调用的工具名称和目的。

> **示例**：
> “请先用 `Glob` 查找 `src/` 目录下所有的 `.cpp` 文件，然后用 `ReadFile` 逐一遍历这些文件，查找其中是否包含 `deprecated` 标记。最后将结果整理到一个新的 `report.md` 文件中，使用 `WriteFile` 写入。”

**效果**：agent 会按照你指定的工具链一步一步执行，路径清晰，可控性强。

### 2. 目标驱动式（Goal-Oriented）

不指定具体工具，而是描述期望达成的目标，让 agent 自主决策。

> **示例**：
> “我想知道这个项目的入口函数在哪里，以及它依赖了哪些第三方库。请自行搜索项目代码并给出结论。”

**效果**：agent 会自主组合 `Glob` + `Grep` + `ReadFile` 等工具进行调研。这种方式更适合探索性任务。

### 3. 约束条件式（Constrained Execution）

在提示词中加入显式约束，限制 agent 的行为边界。

> **示例**：
> “请修改 `config.py` 中的 `MAX_RETRIES` 值为 `5`。要求：
> 1. 必须使用 `StrReplaceFile` 进行最小化修改，不要覆盖整个文件；
> 2. 修改前先用 `ReadFile` 查看当前内容，确认行号；
> 3. 修改后再次用 `ReadFile` 验证结果。”

**效果**：约束条件能有效避免 agent 因“图省事”而采取不够优雅的操作（如直接覆盖文件）。

### 4. 分步工作流式（Step-by-Step Workflow）

对于复杂任务，将步骤拆解并在每一步提示中嵌入工具调用要求。

> **示例**：
> “我们需要为项目添加一个新的 CLI 命令。请按以下步骤执行：
> 1. **调研**：用 `Glob` 和 `ReadFile` 查看现有 CLI 命令的实现方式；
> 2. **计划**：如果涉及多文件改动，请先调用 `EnterPlanMode` 制定修改方案；
> 3. **实现**：编写新命令的实现代码，使用 `WriteFile` 或 `StrReplaceFile`；
> 4. **验证**：运行 `Run` 执行相关测试，确认功能正常；
> 5. **收尾**：用 `SetTodoList` 更新任务进度为全部完成。”

### 5. 元提示词式（Meta-Prompting）

如果你正在设计一个系统提示词（System Prompt），可以在其中固化 agent 使用工具的通用准则。

> **示例**（可作为 system prompt 的一部分）：
> “你是一个精通工具使用的 coding agent。在执行任务时，请遵循以下原则：
> - **先观察，后动手**：修改任何文件前，优先使用 `ReadFile` 或 `Grep` 确认上下文；
> - **最小化变更**：编辑文件时优先使用 `StrReplaceFile`，避免不必要的全文件覆盖；
> - **异步管理**：对于可能超过 30 秒的命令，使用 `Run` 的后台模式并结合 `TaskOutput` 监控；
> - **复杂任务先计划**：涉及 3 个以上文件的改动或架构决策时，必须调用 `EnterPlanMode`；
> - **善用子代理**：当子任务可以独立执行且不影响主流程时，使用 `Agent` 并行处理。”

## 最佳实践与典型场景

### 场景 A：安全地重构一个函数

```text
用户：请将项目中的所有 `print_error` 调用替换为 `log_error`。

Agent 应执行：
1. Grep 搜索 "print_error"，确定所有出现位置；
2. ReadFile 查看关键文件，确认是否有同名局部变量或字符串误匹配；
3. 对每个确认的文件使用 StrReplaceFile 进行替换；
4. 替换后 Grep 再次验证是否还有残留；
5. Run 运行项目测试，确认没有破坏功能。
```

### 场景 B：处理一个交互式命令

```text
用户：请帮我运行这个交互式脚本 `setup.py` 并完成安装向导。

Agent 应执行：
1. Run 启动 `python setup.py`，设置 run_in_background=true；
2. TaskOutput 观察脚本输出的提示问题；
3. Input 向进程发送对应的回答；
4. 循环步骤 2-3 直到安装完成；
5. TaskOutput 最终确认成功信息。
```

### 场景 C：引入一个需要多文件修改的新功能

```text
用户：请为这个项目添加对 JWT 认证的支持。

Agent 应执行：
1. EnterPlanMode 进入计划模式；
2. Glob + ReadFile 调研现有认证中间件结构；
3. 撰写计划文件，调用 ExitPlanMode 展示方案；
4. 用户确认后，按步骤修改多个文件（StrReplaceFile / WriteFile）；
5. SetTodoList 跟踪各子任务状态；
6. Run 运行测试验证。
```

### 场景 D：将外部文档纳入分析

```text
用户：请阅读附件中的设计文档，并检查代码是否实现了文档中的所有要求。

Agent 应执行：
1. 若附件为 PDF，使用 Pdf2md 转换；若为 Word，使用 Docx2md 转换；
2. 分析转换后的 Markdown 内容，提取需求清单；
3. Grep + ReadFile 检查代码实现；
4. 对比并输出差异报告。
```

## 总结

内置工具是 coding agent 从“语言模型”进化为“可执行代理”的桥梁。掌握这些工具的特性与适用场景，并在提示词中清晰、合理地引导其使用，是发挥 agent 最大效能的关键。

**核心原则回顾**：

1. **观察先于行动**：`ReadFile` / `Grep` / `Glob` 永远是修改前的第一步。
2. **最小化改动**：`StrReplaceFile` 是编辑文件的首选，`WriteFile` 仅用于创建新文件或明确需要整体重写的情况。
3. **耗时任务异步化**：善用 `Run` 的后台模式 + `TaskOutput` / `Input` 管理交互与长时任务。
4. **复杂任务先规划**：涉及多文件、架构变更时，务必要求 agent 进入 `EnterPlanMode`。
5. **文档与代码并重**：通过 `Pdf2md`、`Docx2md`、`FetchURL` 将外部信息无缝整合到工作流中。

通过刻意地在提示词中植入这些原则，你将能够让 agent 更加可靠、高效、可控地完成各类软件开发任务。
